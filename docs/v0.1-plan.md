# MAP Python SDK Port — Plan

## Context

MAP currently ships only as a TypeScript npm package (`@model-action-protocol/core`), at `Open Source/`. Engagement is flat. Neil's read: TS-only is the blocker — most agent dev (LangChain, LlamaIndex, CrewAI, Pydantic AI, MCP server reference impls) lives in Python, so a TS-only "open standard" can't actually become an open standard. The PRD already lists "Python SDK middleware" as planned. This plan ports the protocol to Python with cross-language wire-format conformance, so a ledger written by either impl verifies correctly in the other.

**Outcome:** A `pip install model-action-protocol` package at full surface parity with the TS impl, validated against shared conformance fixtures, and paired with two demo artifacts (Jupyter notebook + FastAPI mini-app) so both ML/agent-dev and backend audiences have a five-minute on-ramp.

---

## Strategic principles

1. **Spec is the source of truth, not TS.** Write a standalone `SPEC.md` that defines the wire format, canonicalization, hash inputs, and verdict schema. Both TS and Python implement the spec. If TS diverges from spec, that's a TS bug — not "Python is wrong." This is the move that turns MAP from "TypeScript library" into "protocol with a TS reference implementation," and it's what makes the artifact citable rather than vendored.
2. **Canonicalization: JCS (RFC 8785).** No bespoke recursive key-sort logic. Use `canonicalize` (npm) and `jcs` (PyPI). "MAP uses RFC 8785 JSON canonicalization" is itself a credibility line — researchers, enterprise compliance, and standards bodies all recognize it. **Cost:** existing TS-generated hashes will change once during the migration. We pay this now while there's no installed base, or we never can.
3. **Frozen, versioned conformance fixtures.** `spec/fixtures/v0.1/` is committed to git, never regenerated. New spec version → new directory (`v0.2/`), old fixtures retained for regression. Both TS and Python tests load the same files. A divergence between impls becomes a fixture diff, not a he-said-she-said.
4. **Idiomatic Python, not transliterated TS.** Native `anthropic` SDK (not a Vercel AI SDK shim). Pydantic v2 (not Zod mimicry). `async def` for I/O with `_sync` convenience wrappers. Decorator API for tools alongside builder functions.
5. **Full surface parity at v0.1.** Match the TS public surface — LearningEngine and Postgres adapter included. LearningEngine in particular ("the ledger IS the training data") is the differentiated pitch; shipping without it weakens the story.

## Build sequence — phased

Phase 0 is one-time at v0.1 freeze. Phases 1-4 run on every PR via CI.

### Phase 0 — Spec foundation (do first, lock before Phase 2)

1. **Draft `spec/SPEC.md`.** Standalone document defining: data model (ledger entry, action record, critic result, reversal schema), canonicalization (JCS / RFC 8785, normative reference), hash input order (`sequence`, `action`, `stateBefore`, `stateAfter`, `parentHash`, `critic` — JCS-encoded then SHA-256), genesis parentHash (`"0" * 64`), verdict enum (`PASS | CORRECTED | FLAGGED`), status transitions (`ACTIVE → ROLLED_BACK`), rollback semantics, fingerprint algorithm for LearningEngine (`SHA-256(verdict:tool:correctionTarget)`). Read TS source for behavior, but **specify normatively** — TS quirks become candidates for spec inclusion or TS bug reports, decided on the merits.

2. **Migrate TS to JCS.** Replace `Open Source/src/snapshot.ts:30-41` (`serializeState` recursive sort) with the `canonicalize` npm package. Re-run TS test suite — existing tests will fail because hashes change. Update test expectations. This is intentional and one-time. Document the breaking change in `CHANGELOG.md` (pre-1.0, semver minor bump).

3. **Generate frozen fixtures from migrated TS.** Add a TS script `scripts/generate-fixtures.ts` that produces fixtures meeting the Phase 0 coverage requirements (see Verification gates): semantic cases (PASS-only chain, CORRECTED with reversal, FLAGGED-halt, ROLLED_BACK + resume, LearningEngine patterns) and edge cases (unicode, deeply nested, empty payload, large payload). Write to `spec/fixtures/v0.1/*.json` and **commit to git**. Never regenerate within v0.1. If a future spec version requires new fixtures, create `v0.2/`. Add a TS test that reads `spec/fixtures/v0.1/` and verifies them — this protects against unintended TS regressions.

4. **Manual SHA-256 spot validation.** While drafting the spec, hand-compute the JCS bytes and SHA-256 hash of one fixture entry. Confirm it matches both the spec's worked example and the fixture's stored `hash` field. After this passes once, the validation lives as `test_canonical_json_known_hash` (Phase 1) — hard-coded entry, hard-coded hash, regression gate.

5. **Freeze Phase 0 (ceremony).** Once steps 1–4 are green:
   - Append a `## v0.1 spec frozen — YYYY-MM-DD` section to `Open Source/CHANGELOG.md` summarizing what's locked: spec version, fixture set (commit hash), JCS canonicalization, JS + Python conformance test counts.
   - Create the git tag `spec-v0.1.0` annotated with the same summary. Local-only by default — pushing the tag is a Neil decision (it becomes the citable artifact in external communication).
   - Bump the prose anywhere "MAP, in development" appears to "MAP v0.1 spec, frozen." Update `Open Source/README.md` link to the tagged spec when convenient.

### Phase 1 — Python implementation (against the spec, not the TS code)

Interleave with Phase 0 (steps 1–4) — write Python skeleton to surface spec gaps, then lock the spec. Phase 0 step 5 (freeze ceremony) happens once everything below is otherwise stable.

5a. **Document Python design choices (`python/DESIGN.md`).** Before module work begins, capture the implementation decisions that constrain everything that follows: schema layer (Pydantic v2 with `ConfigDict(extra="forbid", populate_by_name=True)`, `model_dump(exclude_none=True)` for hash payloads), store interface (`Protocol` + `runtime_checkable`, not `ABC`), I/O model (`async def` for all I/O, `_sync` convenience wrappers on the `MAP` class for notebooks), error hierarchy (one `MAPError` base, subclassed for `LedgerStoreError`, `CriticError`, `ToolValidationError`), driver choices (`aiosqlite` for SQLite, `asyncpg` primary + `psycopg[async]` fallback for Postgres). 9. **Port `tools.py`.** Pydantic `BaseModel` subclasses as input schemas. Builder functions + decorator API (`@map.tool(...)`). RESTORE, COMPENSATE, ESCALATE.

10. **Port `executor.py` + `rollback.py` + `learning.py` + `map.py`.** Direct translations from spec. LearningEngine pattern fingerprint must match Phase 0 fixture exactly.

### Phase 2 — Cross-language conformance

11. **Python verifies TS-written fixtures.** `pytest tests/test_conformance.py::test_verify_v01_fixtures` — every `spec/fixtures/v0.1/*.json` is loaded, chain verified, expected `valid: True`.

12. **Python writes new ledgers, TS verifies.** Python script generates fresh ledgers, exports JSON. TS test (`Open Source/__tests__/python-output-conformance.test.ts`) loads them, runs `verifyIntegrity()`, expects `valid: true`.

(b) Python writes intentionally-corrupt ledgers (wrong parentHash, broken sequence, mutated state hash); TS detects each at the correct index. (c) Same shape in TS for Python-written corrupt ledgers.

14. **LearningEngine fingerprint parity.** Python loads `spec/fixtures/v0.1/learning-patterns.json`, runs `analyze_patterns()`, asserts each pattern's `fingerprint` equals the TS-generated value byte-for-byte.

15. **Schema parity.** Diff the DDL produced by `PostgresLedgerStore.init()` in TS vs Python. Identical column types, constraints, indexes. Test asserts no diff.

16. **Write parity (TS → Python).** TS writes 5 entries. Python `PostgresLedgerStore.get_entries()` reads them, exports JSON, hash-verifies against original.

17. **Read parity (Python → TS).** Python writes 5 entries. TS reads, exports, hash-verifies. Catches issues like timestamp precision, JSONB encoding, UUID format mismatches.

20. **SDK integration test — mocked** (`tests/test_sdk_integration.py`). One canonical scenario: MAP wraps an Anthropic tool-use loop. ## Critical implementation details

### JCS canonicalization (RFC 8785)

Both impls use library implementations of JCS. No bespoke recursive sorting.

```python
# Python — model_action_protocol/snapshot.py
import jcs  # https://pypi.org/project/jcs/
import hashlib

def canonical_bytes(obj: Any) -> bytes:
    return jcs.canonicalize(obj)

def sha256_hex(obj: Any) -> str:
    return hashlib.sha256(canonical_bytes(obj)).hexdigest()
```

```typescript
// TS — Open Source/src/snapshot.ts (post-migration)
import { canonicalize } from "canonicalize";  // https://www.npmjs.com/package/canonicalize
import { createHash } from "crypto";

export function canonicalBytes(obj: unknown): string {
  return canonicalize(obj);
}

export function sha256Hex(obj: unknown): string {
  return createHash("sha256").update(canonicalBytes(obj)).digest("hex");
}
```

JCS handles the edge cases that bit homemade canonicalization: number representation (no scientific notation, integer/float distinction), unicode normalization (NFC), key sorting (lexicographic on UTF-16 code units), null vs absent, whitespace. Specifying "MAP uses RFC 8785" replaces a 2-page edge-case appendix in `SPEC.md`.

Verification: `compute_entry_hash(...)` Python output equals pre-recorded TS hash byte-for-byte for every entry in every `spec/fixtures/v0.1/*.json`. Same the other way.

### Pydantic ↔ Zod mapping

- Zod `z.enum(["PASS", "CORRECTED", "FLAGGED"])` → Pydantic `Literal["PASS", "CORRECTED", "FLAGGED"]`
- Zod `.optional()` → `field: Optional[T] = None` with `model_config = ConfigDict(exclude_none=True)` for serialization
- Zod `.refine()` → Pydantic `@field_validator` or `@model_validator`
- Critical: when serializing `LedgerEntry` for hashing, exclude `None` fields to match TS behavior where `undefined` keys don't appear in `JSON.stringify` output.

### Async + sync surface

Public API is `async def` everywhere I/O happens (`map.execute()`, `ledger.append()`, `store.append()`). Provide sync wrappers on `MAP`:

```python
class MAP:
    async def execute(self, goal, tool_name, input): ...
    def execute_sync(self, goal, tool_name, input):
        return asyncio.run(self.execute(goal, tool_name, input))
```

This unblocks notebook/script users without forcing them into async.

### LLM critic — native anthropic SDK

Don't port `generateText`. Take a client directly:

```python
from anthropic import AsyncAnthropic

def create_llm_critic(client: AsyncAnthropic, model: str = "claude-sonnet-4-6") -> CriticFunction:
    async def critic(action, state_before, state_after):
        response = await client.messages.create(
            model=model,
            max_tokens=1024,
            system=CRITIC_SYSTEM_PROMPT,
            messages=[{"role": "user", "content": format_critic_prompt(action, state_before, state_after)}],
        )
        return CriticResult.model_validate_json(response.content[0].text)
    return critic
```

Use Anthropic's native tool-use or structured output for typed verdicts. This is the Python port's "upgrade" — Python users get a first-class Claude integration, not a wrapped abstraction.

## Python design decisions (locked)

These are locked by `python/DESIGN.md` (Neil approved 2026-04-25). Deviations during implementation get back-pressure against the doc.

| # | Question | Decision |
|---|----------|----------|
| 1 | Data modeling | Pydantic v2 (`>=2.5,<3`). No v1 support. |
| 2 | **Sync vs async (refined)** | **Sync all the way down at v0.1.** Public API sync. Internal I/O sync — no event loops, no thread pools, no `asyncio.run()` anywhere in the library. `LedgerStore` Protocol is sync; v0.2 ships sibling `AsyncLedgerStore` and `AsyncMap` (not a replacement). **Critic and reverser are sync `Callable`s at v0.1** (the v0.2 async siblings are separate types). **Architectural constraint:** I/O lives only in store classes — canonicalization, hashing, critic, and reversal logic are pure CPU. This is the constraint that makes async retrofit clean. FastAPI demo uses `asyncio.to_thread()` at call sites with explicit comments. |
| 3 | Interfaces | `typing.Protocol` with `@runtime_checkable`. No ABCs for public interfaces. |
| 4 | Error hierarchy | Layered, single `MapError` root: `ValidationError`, `LedgerError` (`LedgerCorruption`, `EntryNotFound`, `StoreError`), `ReversalError` (`NotReversible`, `ReversalFailed`), `CriticError`, `ConformanceError`. No stdlib multi-inheritance. |
| 5 | **Package layout** | **Top-level `map/`** with `core/`, `stores/`, `learning/`, `tools/`, `integrations/`. Public classes: `Map`, `Action`, `Ledger`, `LedgerEntry`, `Reversal`, `MapError`. PyPI: `model-action-protocol`. `py.typed` shipped. |
| 6 | Testing | pytest + `hypothesis` + markers (`conformance`, `integration`, `perf`, `postgres`, `live_api`). Phase 1 filter: `pytest -m "not conformance and not integration and not perf and not postgres and not live_api"`. |
| 7 | Logging | stdlib `logging`, namespaced under `map.*`, `NullHandler` default. |
| 8 | Type checking | `mypy --strict` floor in CI. Pyright recommended. |
| 9 | Python versions | 3.10–3.13. `requires-python = ">=3.10,<3.14"`. |
| 10 | **Dependencies** | Pydantic v2 only at core. **Vendor JCS** (~50 LOC in `map/core/canonical.py`); no `jcs` PyPI dep. Postgres/Anthropic/FastAPI as extras. |
| 11 | Style | ruff for lint + format + import sort. No black, no isort. |
| 12 | Versioning | SemVer for the impl. Separate `SPEC_VERSION = "0.1"` constant alongside `__version__`. |
| 13 | **Anthropic integration** | Thin wrapper. One primitive: `wrap_tool_call(map_instance, tool_use_block, registry)` plus `@m.reversible(reverser=...)` decorator. Not building a framework. |

### Refactor required (before continuing Phase 1)

The current Python skeleton (`model_action_protocol/`) was built async-first against decisions that have now been overridden. The locked doc forces the following before module work continues:

1. **Sync-all-the-way-down conversion.** `Ledger.append`, `LedgerStore.*`, `InMemoryLedgerStore.*` go from `async def` → `def`. Critic and reverser callables become sync `Callable`s. **Architectural enforcement:** audit `ledger.py` to confirm zero I/O outside the `store.*` calls. Tests update to drop `await`/`@pytest.mark.asyncio`. ~2 hours.
2. **Rename + restructure** `model_action_protocol/` → `map/` with the documented nested layout (`core/`, `stores/`, `learning/`, `tools/`, `integrations/`, `exceptions.py`). Class renames: `MAP` → `Map`, `ActionRecord` → `Action`, `CompensatingAction`/`CriticCorrection` flatten into nested types. ~1 hour.
3. **Vendor JCS** in `map/core/canonical.py`. Replace `jcs` PyPI dep with ~50 LOC. Conformance fixtures (which we already verified byte-for-byte against the `jcs` package) become the regression suite for the vendored impl. ~1 hour.
4. **Layered exception hierarchy** in `map/exceptions.py`: `MapError` → `ValidationError`, `LedgerError` (`LedgerCorruption`, `EntryNotFound`, `StoreError`), `ReversalError` (`NotReversible`, `ReversalFailed`), `CriticError`, `ConformanceError`. ~30 min.
5. **Logging setup** with `NullHandler` defaults and namespaced loggers (`map`, `map.ledger`, `map.critic`, `map.stores.postgres`, `map.integrations.anthropic`). ~20 min.
6. **`py.typed` marker** (one empty file). ~1 min.
7. **Bump Python floor** in `pyproject.toml` from `3.11` to `3.10`. Verify nothing relied on 3.11-only features. ~10 min.
8. **PyPI distribution name** in `pyproject.toml`: `model-action-protocol` → `model-action-protocol`. ~5 min.

Total: roughly half a day of refactor before Phase 1 module work resumes (critic, executor, rollback, learning, sqlite store, postgres store, the `Map` orchestrator).

## Verification — gated by phase

The plan distinguishes **definition-of-done** (what must be true to call something v0.1) from the **CI runbook** (what runs on every PR). They overlap but are not identical.

### CI cadence

| Cadence | Gates |
| --- | --- |
| **Every PR** | Phase 1, Phase 2, Phase 4 mocked SDK integration test |
| **Every PR if `DB_HOST` is set, else nightly** | Phase 3 (Postgres parity) |
| **Nightly** | Phase 4 notebook + FastAPI smoke + live SDK test (gated on `ANTHROPIC_API_KEY`) |
| **One-time / release-only** | Phase 0, Phase 5 |

This keeps PR feedback under a few minutes without losing coverage.

**"Nightly" means a scheduled GitHub Actions workflow** (`.github/workflows/nightly.yml`, `on: schedule: cron: "0 7 * * *"` UTC = midnight Pacific). Concurrency policy: `concurrency.group: nightly`, `cancel-in-progress: false` so an in-flight run finishes before the next starts; report step uses `if: always()` so partial results upload even on failure. Required CI secrets: `DB_HOST`/`DB_USER`/`DB_PASSWORD` for the Postgres job, `ANTHROPIC_API_KEY` for the live SDK job. Setting these up is its own ticket — until both are in place, the nightly gates are aspirational and the corresponding tests `skip` cleanly.

### Phase 0 gates (one-time, at v0.1 freeze)

- **Canonicalization decision recorded.** Spec selects an approach (default: JCS / RFC 8785) and explicitly specifies handling for: number representation (no scientific notation, integer/float distinction), Unicode normalization (NFC), key ordering (lexicographic on UTF-16 code units), null vs absent fields (omit absent), whitespace (none). If MAP layers anything on top of the chosen library (e.g., it does not), the spec calls that out.
- **Fixture coverage requirements met.** `spec/fixtures/v0.1/` contains at least one fixture for each of: (a) PASS-only multi-entry chain, (b) CORRECTED action with reversal schema, (c) FLAGGED action that halts, (d) ROLLED_BACK + resume, (e) LearningEngine pattern fingerprints, and (f) edge cases — unicode strings (NFC vs NFD inputs converging), deeply nested object, empty payload, large payload (~10KB serialized). Any future gap becomes a new fixture.
- **TS test suite passes post-JCS migration.** `cd "Open Source" && npm test`. Hashes change vs 0.1.x; this is intentional.
- **Spec worked example pinned.** `spec/SPEC.md` §6.4 contains a hand-computed entry hash that an automated test on each side asserts equality against.
- **Conformance fixtures committed and verified by TS.** `npm run test:fixtures` passes; the directory is treated as immutable thereafter. **Immutability enforcement is honor system at v0.1.** A future CI guard (suggested: `git diff --name-only origin/main HEAD | grep -q "spec/fixtures/v0.1" && exit 1` unless the spec version bumps) is the cheap mechanical version when honor system breaks.

### Phase 1 gates (Python unit, every PR)

- `cd "Open Source/python" && pytest -m "not conformance and not perf and not postgres and not live_api"` — snapshot, ledger, map, critic, tools, persistence (SQLite), learning all green. **The mocked SDK integration test (`test_sdk_integration.py`) runs in Phase 1; it is unmarked by design. There are no other marker-based exclusions** — if a future test file is marked `integration`, that marker has no effect at this stage and the test still runs. Marker exclusions exist only for tests that are slow, environment-gated, or release-only (`conformance`, `perf`, `postgres`, `live_api`).
- `test_canonical_json_known_hash` passes — hard-coded entry → hard-coded hash from spec §6.4. **Each impl ships its own copy of this test** (Python: `tests/test_snapshot.py`; TS: `src/__tests__/conformance.test.ts`). Failure on either side indicates spec drift or canonicalization regression and blocks merge.

### Phase 2 gates (cross-language conformance, every PR)

- **Forward direction.** `pytest tests/test_conformance.py::test_verify_v01_fixtures` — Python verifies every `spec/fixtures/v0.1/*.json`, expects `valid: True`.
- **Reverse direction.** `npm run test:python-output-conformance` — TS loads Python-written ledgers, runs `verifyChain()`, expects `valid: true`.
- **Negative tests, both directions, enumerated.** For each fixture and for live-generated ledgers, both impls detect each of these and report the correct `corruptedAt` index:
  1. **Field tampering** — change a value inside an entry's `action` or `critic`.
  2. **Hash tampering** — replace a stored `hash` field with a wrong value.
  3. **Order tampering** — swap two consecutive entries.
  4. **Insertion** — synthesize and inject a new entry between N and N+1.
  5. **Deletion** — remove entry N from the chain.
- **LearningEngine fingerprint parity.** Python's `analyze_patterns()` over `learning-patterns.json` produces fingerprints byte-identical to TS-recorded values.
- **Unicode normalization sensitivity.** Both impls assert that NFC and NFD encodings of the same semantic string produce **different** state hashes (i.e., neither impl silently normalizes). Spec §5 mandates JCS without normalization; this test pins the behavior so users can't accidentally rely on it.

> **Truncation is explicitly not detected.** Dropping the tail leaves a valid shorter chain; detection requires external state (transparency log, signed latest-hash, etc.). v0.1 declares this in spec §10 and out-of-scope below.

### Phase 3 gates (Postgres parity, every PR if `DB_HOST` set, else nightly)

- **Schema parity via introspection** (not raw DDL string equality). Both impls run their migration against a fresh Postgres database. Both then query `information_schema.columns` and `pg_indexes` (or `pg_catalog.*` equivalents). The normalized schema — column name, normalized type, nullability, default, indexes — **MUST** be identical. Raw DDL is allowed to differ (e.g., `VARCHAR(255)` vs `TEXT` is fine if both produce equivalent columns).
- **Write parity (TS → Python).** TS writes 5 entries; Python `PostgresLedgerStore.get_entries()` reads them; round-tripping through Python's serialization preserves all fields and hashes verify.
- **Read parity (Python → TS).** Python writes 5 entries; TS reads them; same round-trip + hash verification. Catches timestamp precision, JSONB encoding, and UUID format mismatches that schema parity wouldn't.

### Phase 4 gates

**Every PR:**
- `pytest tests/test_sdk_integration.py` — MAP wraps mocked Anthropic tool-use loop; ledger entry verifies; reversal works.

**Nightly (slow, network-dependent):**
- `jupyter nbconvert --execute python/examples/quickstart.ipynb` runs cleanly.
- `cd python/examples/fastapi_app && uvicorn main:app` + scripted `curl` of all four endpoints returns expected shapes.
- `pytest -m live tests/test_sdk_integration_live.py` (gated on `ANTHROPIC_API_KEY`; skip cleanly if absent). Catches real-world quirks the mocked test won't — tool-use response shape drift, streaming edge cases, rate-limit handling. Mocks pass when MAP wraps your mock; live test fails when the world surprises you.

### Phase 5 gates (release-only)

- `python -m build` produces a clean wheel; `twine check dist/*` passes.
- **Clean-venv quickstart, matrix.** From a fresh `python -m venv .quickstart-test` with no MAP-related deps installed, the README's quickstart commands install the published package and produce the documented output. Run on **both Linux and macOS CI runners** — psycopg's binary wheel and SQLite extension paths can diverge between platforms, and the researcher audience is heavily macOS. If a Windows runner is cheap to add, do so; otherwise document v0.1 as "Linux + macOS tested, Windows expected to work." (Cost note: GitHub-hosted macOS minutes bill at 10× Linux. Acceptable for a release-only gate; **do not** promote this to nightly.)
- `npm publish --dry-run` clean for the migrated TS package.
- Spec §6.4 worked example computes correctly under the released wheel + the released npm tarball.

## Out of scope (intentionally)

The spec is the source of truth for protocol-level out-of-scope. **`spec/SPEC.md` §10.2 is authoritative** for: cryptographic key management, tail-truncation detection, and multi-agent identity verification. This plan does not restate those — see the spec.

This section covers **implementation/process** out-of-scope only:

- **Python-side Vercel AI SDK shim.** Python users have `anthropic`, `openai`, `litellm`. No need to wrap.
- **Async stores at v0.1.** v0.1 ships sync `LedgerStore`, `SQLiteStore`, `PostgresStore` only (per locked design doc §2). v0.2 will add `Async*Store` siblings using `psycopg[async]` (same library as v0.1's sync `psycopg[binary]`, no second migration). Wire format identical.
- **Perf gates in CI.** Smell-test only, runs on releases via `pytest -m perf`. We're not optimizing for throughput at v0.1.
- **Multi-impl repo split.** Spec + both impls colocated for v0.1. Splitting into separate repos (e.g., `map-spec`, `map-ts`, `map-py`) is deferred until contributor friction warrants it.
